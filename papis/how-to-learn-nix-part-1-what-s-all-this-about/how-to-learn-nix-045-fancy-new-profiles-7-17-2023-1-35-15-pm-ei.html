Okay, so I actually saw an option yesterday, when I was scanning through man 5 nix.conf, but I didn‚Äôt mention it at the time because I was busy with other things. But I believe that it‚Äôs relevant here:

So obviously‚Ä¶ let‚Äôs not do that. I download the file locally, and add this line to my ~/.config/nix/nix.conf:

Heavy sigh. I guess that ~ expansion is a feature of my shell, so it was foolish of me to expect Nix to parse that as a local file path. Is that really a valid, like, URI? I dunno.

Let‚Äôs try the absolute version:

$ grep flake-registry ~/.config/nix/nix.conf

$ nix path-info 'nixpkgs#git'
/nix/store/bcjf7pr923hinmzcci9qh47g4zd3ds4q-git-2.34.0

Okay, that worked fine.

So. I can now‚Ä¶ query properties of my Nix store again, without making HTTP requests to GitHub.

Ha. Without making these particular HTTP requests to GitHub. I‚Äôve already squashed two different cases where Nix was connecting to The Internet in order to run basic commands that previously worked offline. Who knows how many more of those there are.

Anyway, nix path-info works; it‚Äôs slightly worse, because I have to type the quotes and the # character and all that, but whatever. It‚Äôs not a command I use very often.

Now, you might reasonably be wondering at this point: is this real? Did GitHub just happen to be down at the one moment you started writing this blog post? Or did you stage all of this just for rhetorical effect?

That‚Äôs a good question. The answer is this 100% happened; this is all real, but that it‚Äôs not because GitHub is down.

I had GitHub blackholed in my /etc/hosts because it‚Äôs on a long list of tech-related websites that I disable when I am trying not to think about computer things for a moment, just one moment, just one brief second of quietude, and I forgot to re-enable it when I started writing this blog post.

And I‚Äôm glad I did, because I learned‚Ä¶ well, that Nix is constantly making HTTP requests to GitHub!

I once again wish that I had dtruss available, so I could see what else Nix is doing. I can‚Äôt imagine what it must feel like to run Nix without a fast internet connection right now.

Anyway, we were supposed to look at the new flake-enabled profiles, right?

Let‚Äôs do that.

And then maybe I can figure out how to install my sd overlay.

nix profile --help

So we have all of the following subcommands:

¬∑ nix profile diff-closures - show the closure difference between each version of
  profile
¬∑ nix profile history - show all versions of a profile
¬∑ nix profile install - install a package into a profile
¬∑ nix profile list - list installed packages
¬∑ nix profile remove - remove packages from a profile
¬∑ nix profile rollback - roll back to the previous version or a specified version
  of a profile
¬∑ nix profile upgrade - upgrade packages using their most recent flake
¬∑ nix profile wipe-history - delete non-current versions of a profile

I‚Äôm excited about upgrade, as that was a pretty big missing feature of nix-env.

I learn that it uses the same numbered-symlink approach that nix-env did, and that the default profile is still in ~/.nix-profile, and it still supports --profile to change that.

I learn that the manifest.nix has been replaced by manifest.json, and that its contents is actually documented now. Nice! That‚Äôs great. No more pawing around in the dark trying to figure out just how to look at that file.

And that‚Äôs all that the top-level docs have to say. Let‚Äôs dive in!

$ tree ~/scratch/test-profile

Neat. Okay. Same as before, basically, with a slightly nicer looking command. Except instead of nixpkgs.hello ‚Äì a reasonable, intuitive syntax ‚Äì it‚Äôs now 'nixpkgs#hello', because‚Ä¶ because, really, I don‚Äôt even know, you know?

It‚Äôs not like it was pleasant to type nix-env -iA nixpkgs.hello before, and indeed I made an alias so I could just type sd nix install hello a long time ago. I expect I‚Äôll do the same for all of the common flake commands I type, given time.

‚îú‚îÄ‚îÄ manifest.json
‚îî‚îÄ‚îÄ share -> /nix/store/6sv1isax4axkxfnmg4gxg1pzr4417r6v-hello-2.10/share

Huh.

Did the remove secretly fail?

$ nix profile remove 'nixpkgs#hello' --profile ~/scratch/test-profile

$ echo $?

Yep. Just a silent no-op. Cool.

Well, let‚Äôs see. nix profile remove --help lists all of the following possible invocations:

Well‚Ä¶ none of those are what I want. What are package ‚Äúpositions‚Äù? I don‚Äôt know. Presumably I could find out with --list:

$ nix profile list --profile ~/scratch/test-profile
0 flake:nixpkgs#legacyPackages.x86_64-darwin.hello github:NixOS/nixpkgs/18e6b5184274305f2c9dc36141003473375a5df9#legacyPackages.x86_64-darwin.hello /nix/store/6sv1isax4axkxfnmg4gxg1pzr4417r6v-hello-2.10

Yeah, okay, I assume that‚Äôs what the 0 means. But‚Ä¶ what? This seems like some weird implementation detail leaking out. I‚Äôm not supposed to actually know‚Ä¶ what?

So‚Ä¶ let me get this straight. If I‚Äôm reading this correctly: in order to remove a package that I installed as nixpkgs#hello, I‚Äôm supposed to type:

$ nix profile remove packages.x86_64-darwin.hello

Is that‚Ä¶ is that real? Is that the actual CLI?

I guess those numbers are there as like‚Ä¶ ergonomic aids? Because the designer of this feature realized that it was unreasonable to expect someone to type packages.x86_64-darwin.hello, so instead of supporting the obvious invocation‚Ä¶?

Remember when I was excited that nix profile would liberate me from the terrible UI of nix-env? How young I was, then.

Anyway let‚Äôs try it out anyway:

‚îî‚îÄ‚îÄ share -> /nix/store/6sv1isax4axkxfnmg4gxg1pzr4417r6v-hello-2.10/share

Fantastic.

Ah, because by spooning my way through the punctuation soup of the nix profile list output, I guess that I‚Äôm actually supposed to type:

$ tree ~/scratch/test-profile
/Users/ian/scratch/test-profile
‚îî‚îÄ‚îÄ manifest.json

Okay, well, that was horrible; let‚Äôs never do that again.

I basically never run nix-env --uninstall, because I use nix-shell -p pretty frequently, and only add something to my profile if I‚Äôm pretty confident that I‚Äôm going to keep it around. And even when I do uninstall something from my profile, I usually do that by typing sd nix sync.

So let‚Äôs try to come up with the equivalent flake-based declarative profile.

So just to refresh, I currently have a file that looks like this:

$ cat ~/dotfiles/user.nix
with import <nixpkgs> {}; [
  git
  nix
  zsh
  ...
]

And I run nix-env -irf ~/dotfiles/user.nix to set my profile to only the derivations listed in that file. I use this to install things and to uninstall things, and also to upgrade things, because previously Nix didn‚Äôt have a way to upgrade things.

So: how do I do that with profiles?

Well, gosh. From reading the --help pages, it doesn‚Äôt seem like there‚Äôs a way to both add and remove packages from a profile in a single operation.

I could do something where I like‚Ä¶ uninstall everything, and then install a list of things. But that would mean that nix profile rollback wouldn‚Äôt revert my ‚Äúsync‚Äù operation; it would revert the install, and then it would revert the uninstall everything.

So‚Ä¶

How am I supposed to write down the packages that I want in my profile?

Another very important part of my workflow is sd nix diff, which shows me output like this:

$ sd nix diff
- htop 3.1.0
+ racer 2.1.48
! nodejs 14.18.0 -> 14.18.1
! git 2.33.0 -> 2.33.1
! delta 0.8.3 -> 0.10.2

So I can see what effect running sd nix sync will have on my profile.

But‚Ä¶ well, I guess I can sort of maybe figure out the ‚Äúcurrent version‚Äù by running nix profile list, and parsing the output? There are no options to control what that actually prints out. There‚Äôs no equivalent of nix-env -q --json, which is what I use to generate that diff.

Okay no. I should have paid attention before. The old manifest.nix was a full working copy of the derivation. This is just‚Ä¶ well, I can probably turn those into derivations?

$ nix flake show 'nixpkgs#hello'
error: unexpected fragment 'hello' in flake reference 'nixpkgs#hello'

I don‚Äôt actually know what to do with a flakeref once I have it. I don‚Äôt know how to evaluate one, or to see anything about the derivation that it evaluates to. Huh.

I look through nix --help and on a whim I tried nix eval.

And it just‚Ä¶ hard broke?

So no idea‚Ä¶ what that‚Äôs about.

nix eval --help gives this example:

I copy it, foolishly:

And I sigh deeply, and run:

Okay. So that works? But I don‚Äôt really want the path. I want the pname and version.

Okaaaay. Oh, I should maybe mention: those \ characters are added by my shell, to tell me that the command actually had no newline after it, so zsh inserted it for me. On bash, this would instead look like:

Anyway. I can evaluate one of these things, but how do I evaluate both of these things? I do not know. I tried a few different invocations, but none of them worked. So I gave up.

I want to do this so that I can diff the ‚Äúcurrent‚Äù version to the ‚Äúnew‚Äù version before I run upgrade. And here I‚Äôm just trying to see the ‚Äúcurrent‚Äù version. But how do I see the new version?

Let‚Äôs see. I look in nix profile upgrade --help, expecting to see a --dry-run flag or something. But there isn‚Äôt one. So I don‚Äôt know how to see the new version either.

So: my conclusion from all of this is that nix profile is not a viable replacement for nix-env. There‚Äôs no way to print a human-readable list of the currently installed packages, there‚Äôs no way to see what is going to be upgraded, there‚Äôs no way to‚Ä¶ use nix profile.

So I won‚Äôt!

Which is fine. It is an experimental feature, after all. I can keep using nix-env.

The reason to use nix profile is, of course, that nix search now operates in flake town, and I don‚Äôt want to nix search through a slightly different set of things than I will be able to install.

So let‚Äôs instead try to figure out how to fix that, and to make nix search match nix-env again. To make a ‚Äúflake‚Äù that matches the contents of my channel.

Hmm. That shouldn‚Äôt be too hard.

$ nix registry add nixpkgs ~/.nix-defexpr/channels/nixpkgs

And now my nixpkgs flake is tied to my channel, and I can update it with normal nix-channel --update commands. Right?
