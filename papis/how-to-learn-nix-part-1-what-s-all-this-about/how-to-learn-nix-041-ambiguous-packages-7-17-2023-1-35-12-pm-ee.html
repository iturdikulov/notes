How to Learn Nix, Part 41:
Ambiguous packages

I opened a PR a few days ago to try to fix the issue I ran into with nix-env -u trying to upgrade to a weird alpha pre-release version of Python.

In my head, I was making the world a better place. It was an issue that bit me; it was an issue that bit someone who emailed me; it seemed like an issue worth fixing. But the maintainer of Python responded to that PR with:

I don‚Äôt think we should be doing this. Yes, it resolves the issue for python3, but there are others out there as well. ‚ÄúIt is known‚Äù that nix-env -i should not be used.

My gut reaction to that comment was: well, sure, but we shouldn‚Äôt let perfect be the enemy of good. python3 is such a common, important package! Who cares if there are some random little packages sprinkled here and there that have the same problem. It‚Äôs worth fixing for big packages like Python.

But this person knows a lot more about Nix than I do. And a lot more about Nixpkgs. So if they believe that this is not worth fixing, there is a very good chance that this is not worth fixing, regardless of what my gut has to say about it.

So: let‚Äôs find out? Let‚Äôs look into this, and figure out how widespread this issue actually is.

I had assumed that it was a rare issue, because python3 is the only problematic package I‚Äôve encountered, and I have quite a few packages installed in my environment at this point. Add to that the fact that it‚Äôs a relatively recent problem ‚Äì introduced some time between NixOS 20.09 and NixOS 21.05 ‚Äì and it seems like this isn‚Äôt typical. python3 is a weird outlier, right?

Let‚Äôs find out!

So I did. I found out. I did the thing. And, having done the thing, I have changed my tune:

python3 is the least of our worries.

nix-env -i is broken for so many packages ‚Äì big, important packages ‚Äì that there‚Äôs no point fixing python3. nix-env -i will never work; nix-env -u will never work; we should give up and go home and focus all efforts on fixing the manual so no one ever runs those commands again. We should honestly just deprecate those commands and print a warning to stderr any time someone runs them that explains how broken they are and why you shouldn‚Äôt run them and what to do instead.1

So: how did I arrive at this conclusion? Let‚Äôs crack open my diary, and find out‚Ä¶

So to do the priority sort, we coalesce null to 0 and then take advantage of this guarantee made by group_by:

group_by(.foo) takes as input an array, groups the elements having the same .foo field into separate arrays, and produces all of these arrays as elements of a larger array, sorted by the value of the .foo field.

Emphasis mine. Without all the insane version stuff:

$ jq <by-name.json '.[] |= (group_by(.priority // 0) | .[0] | sort_by(.version) | .[0].version)'
...

So you can see how the priority check works in isolation. That one‚Äôs pretty simple. Takes a long time to run, though!

    • Actually, I‚Äôm gonna go ahead and close that one right here. I look in the source of nix-env, and find the following comment:

      /* Find the derivation in the input Nix expression
   with the same name that satisfies the version
   constraints specified by upgradeType.  If there are
   multiple matches, take the one with the highest
   priority.  If there are still multiple matches,
   take the one with the highest version.
   Do not upgrade if it would decrease the priority. */

      And looking through the code explains everything we‚Äôre seeing. nix-env -u explicitly looks for something with both 1) a higher version than our installed package and 2) a higher-or-equal priority than our installed package. So even though sqlite-3.27.2+replication3 had a higher priority, it did not have a higher version, so the upgrade skipped it. If there were, say, sqlite-3.99.2+replication3, we would have upgraded to that.

      This makes some sense: it would be weird for an ‚Äúupgrade‚Äù to actually ‚Äúdowngrade‚Äù a package. But my mind deeply expects, on some primordial level, nix-env -u package to be the same as nix-env -e package followed by nix-env -i package. The fact that those are so different is very upsetting to me. But, you know, it‚Äôs not like I‚Äôm ever going to run nix-env -u again, so who cares.

      Reading the code tells me that you can get the behavior I was expecting with nix-env -u --always, which appears to have the same behavior as running nix-env -i for all your packages:

      $ nix-env -u --always --dry-run
(dry run; not doing anything)
downgrading 'imagemagick-7.0.11-9' to 'imagemagick-6.9.12-12'
downgrading 'sqlite-3.35.2' to 'sqlite-3.27.2+replication3'

      --leq
    In addition to upgrading to newer versions, also "upgrade" to
    derivations that have the same version. Version are not a unique
    identification of a derivation, so there may be many derivations
    that have the same version. This flag may be useful to force
    "synchronisation" between the installed and available derivations.
